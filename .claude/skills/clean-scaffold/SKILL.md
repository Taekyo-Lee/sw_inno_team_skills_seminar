# Clean Scaffold

This skill restores the project root to a clean state by removing everything that was generated by scaffold skills (like `headless-agent-ui`). The goal is to get back to a bare repo with only the hand-authored files, so a scaffold can be cleanly re-applied.

## When to use

- Before re-running a scaffold skill to regenerate from scratch
- When the user wants to reset the project to its original state
- After experimenting with a scaffold and wanting to start fresh

## What gets preserved

These files/directories are part of the repo identity and should never be deleted:

| Path | Why |
|------|-----|
| `.git/` | Git history — deleting this destroys the repo |
| `.claude/` | Skill definitions and Claude config |
| `.gitignore` | Git ignore rules (hand-authored) |
| `README.md` | Project documentation |

Everything else in the project root is considered scaffold-generated and should be removed.

## Steps

### Step 1: Stop running processes

Scaffold skills often leave a dev server running in the background (e.g., `npm run dev` which spawns Vite + Express). These processes hold file locks and will interfere with deletion. Kill them first:

```bash
pkill -f "concurrently.*vite.*tsx" 2>/dev/null
pkill -f "tsx watch server" 2>/dev/null
pkill -f "node.*vite" 2>/dev/null
```

Wait a moment, then verify nothing is still running:

```bash
pgrep -af "vite|tsx" | grep -v pgrep || echo "all stopped"
```

If processes persist, use `kill -9` on the remaining PIDs.

### Step 2: Remove scaffold files and directories

Remove all known scaffold artifacts. The typical set for a Node.js/Vite scaffold:

```bash
rm -rf node_modules/ server/ src/ dist/ .vite/ .playwright-mcp/
rm -f package.json package-lock.json tsconfig.json vite.config.ts index.html
```

For safety, list what's left afterward to confirm only preserved files remain:

```bash
ls -la
```

Expected result: only `.git/`, `.claude/`, `.gitignore`, and `README.md`.

### Step 3: Verify with git status

Run `git status --short` to confirm the working directory is clean. The only untracked or modified items should be things that belong in the repo (like new skill definitions or .gitignore changes).

## Adapting for other scaffolds

If a non-Node.js scaffold was used (e.g., Python), adjust the file list accordingly. The principle stays the same: remove everything that isn't in the "preserved" list above. When in doubt, check `.gitignore` — anything listed there as ignored is safe to delete.
