# Gitignore Scaffold

This skill keeps a git repository clean after scaffold skills generate runtime/build files into the project root. The core problem: skills like `headless-agent-ui` copy working app files (package.json, src/, server/, etc.) into a repo that was originally just a README + skill definitions. Without a .gitignore, `git status` explodes with hundreds of untracked files, and someone might accidentally commit node_modules or build artifacts.

## When to use

- Right after a scaffold skill (e.g., `headless-agent-ui`) generates files into the project root
- After `npm install` creates `node_modules/`
- When `git status` shows scaffold-generated files as untracked and they shouldn't be committed
- When the user asks to clean up git tracking or set up .gitignore

## How it works

### Step 1: Assess current state

Before touching anything, understand what's already in place:

1. Check if `.gitignore` already exists in the project root — read it if so
2. Run `git status --short` (never use `-uall` on large repos) to see what's untracked
3. Check `git ls-files` to see if any scaffold files are already tracked — these need special handling

### Step 2: Determine what to ignore

The ignore list depends on which scaffold generated the files. For a typical Node.js/Vite scaffold (like headless-agent-ui), the generated artifacts are:

| Category | Files | Why ignore |
|----------|-------|------------|
| Dependencies | `node_modules/` | Reproducible from package.json, massive |
| Build output | `dist/` | Regenerated on every build |
| Package manifests | `package.json`, `package-lock.json` | Generated by scaffold, not hand-authored |
| Config | `tsconfig.json`, `vite.config.ts` | Generated by scaffold |
| App source | `index.html`, `server/`, `src/` | Generated by scaffold |
| Misc | `*.log`, `.DS_Store` | Never belong in git |

Files that should **always stay tracked**:
- `README.md` — project documentation
- `.claude/` — skill definitions, settings
- `.gitignore` itself — so the rules persist

### Step 3: Create or update .gitignore

**If no .gitignore exists**, create one from the template below.

**If .gitignore already exists**, read it and only append entries that are missing. Don't duplicate existing rules. Add new entries under a clearly labeled section comment.

### Step 4: Handle already-tracked files

If `git ls-files` shows that any of the to-be-ignored files are already tracked, they need to be removed from the index (not from disk):

```bash
git rm --cached <file>           # for individual files
git rm -r --cached <directory>/  # for directories
```

This is important because .gitignore only affects untracked files — it won't ignore files that git is already tracking. But confirm with the user before running `git rm` commands, since this changes the staging area.

### Step 5: Verify

Run `git status --short` after applying changes. The output should show only:
- `.gitignore` (new or modified)
- Any files the user intentionally wants to keep tracked

If scaffold artifacts still show up, something was missed — check the .gitignore rules.

## .gitignore template

```gitignore
# === Scaffold-generated artifacts ===
# Files below are generated by scaffold skills (e.g., headless-agent-ui)
# and should not be committed to the repository.

# Dependencies
node_modules/

# Build output & caches
dist/
.vite/

# Scaffold-generated app files
package.json
package-lock.json
tsconfig.json
vite.config.ts
index.html
server/
src/

# Logs & OS artifacts
*.log
.DS_Store
```

## Adapting for other scaffolds

This template targets Node.js/Vite scaffolds. If a different scaffold generates different files (e.g., a Python scaffold producing `venv/`, `requirements.txt`, `__pycache__/`), adjust the ignore list accordingly. The principle is the same: ignore anything that is generated/reproducible, keep anything that is hand-authored or defines the project identity.
